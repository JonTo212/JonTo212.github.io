Custom Movement System

Rigidbody-based character controller with coyote time, variable jump height, and wall-slide mechanics implemented in C# without Unity's built-in CharacterController. All physics applied in FixedUpdate() for frame-rate consistency.

◈// physics_puzzles.png
Physics-Based Puzzles

Interactive environment objects respond to player momentum and ability use, creating emergent puzzle solutions beyond the designed intent. Rigidbody interactions are layered with trigger zones to keep puzzle logic readable and debuggable.

◈// ability_design.png
Ability Design

Three core abilities — dash, ground pound, and lasso tether — each with a distinct feel and interaction space. All include cooldown feedback, cancel windows, and edge-case handling for mid-air and contact scenarios.

◈// level_architecture.png
Level Architecture

12 hand-crafted levels with difficulty pacing informed by iterative playtesting. Designed using modular tile sets built in Aseprite and imported via Unity's tilemap system. Each level introduces one new mechanic and one new combination.

// overview

Cosmic Ranger V is a physics-driven puzzle platformer developed solo over an eight-month capstone period. The player navigates increasingly complex environments using a momentum-based movement system built entirely from scratch in Unity, without relying on built-in CharacterController components.

The core design challenge was making physics feel expressive and readable — not fighting the player, but rewarding mastery. Every mechanic was iterated on extensively with playtesting feedback incorporated at each milestone.

// deep_dive — frame rate independent movement feel

The most persistent bug in the project wasn't a crash or a logic error — it was that the game felt wrong on machines with different frame rates. On a 144Hz monitor, the player slid further on ice surfaces than intended. On 30fps, jumps felt floaty and imprecise. The movement system was subtly frame-rate dependent without anyone catching it during early development.

problem ◎
Physics running in Update(), not FixedUpdate()

Velocity was being applied each frame in Update() — meaning at 144fps the player received 144 velocity nudges per second vs. 30 at lower frame rates. Friction and drag calculations compounded the inconsistency. The game had shipped to internal testers before the root cause was identified.

approach ◈
Separating input polling from physics application

The fix required splitting the controller into two layers: input state is read and stored in Update() every frame (preserving responsiveness), but all Rigidbody velocity changes are applied only in FixedUpdate(). Jump buffering was added as a side benefit — storing a jump intent for up to 3 frames so inputs just before landing still register.

PlayerController.cs — input/physics separation C# 
private bool  jumpPressed;
private float jumpBufferTimer;
private const float JumpBuffer = 0.1f;

// Update — poll input only, never touch Rigidbody here
void Update() {
    if (Input.GetButtonDown("Jump")) {
        jumpPressed    = true;
        jumpBufferTimer = JumpBuffer;
    }
    jumpBufferTimer -= Time.deltaTime;
    if (jumpBufferTimer <= 0f) jumpPressed = false;
}

// FixedUpdate — all velocity & physics changes go here
void FixedUpdate() {
    ApplyMovement();
    if (jumpPressed && CanJump()) {
        ExecuteJump();
        jumpPressed = false;
    }
}
result ▸
Consistent feel across all tested frame rates

After the refactor, movement behaviour was verified consistent at 30, 60, and 144fps. Jump buffering also meaningfully improved playtest scores on the "responsiveness" axis. The structural lesson — treat Update() as an input bus, FixedUpdate() as the physics executor — became the architecture standard for the rest of the project.
