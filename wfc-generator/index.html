// key_features[]
Wave Function Collapse

A constraint-based Wave Function Collapse system that builds patterns by progressively reducing possibilities across a grid. The grid starts fully “uncollapsed,” with every cell holding a full set of possible tiles. The generator then selects the cell with the lowest entropy (fewest valid options) and collapses it. That choice then triggers a propagation step, where neighbouring cells have their valid options reduced based on adjacency rules, continuing through a queue until the entire grid is resolved. I built the system using ScriptableObjects to define tile data and adjacency rules, using directional enums for face checks to keep it scalable and easy to expand. Tile selection also supports weighted randomness, allowing certain tiles to appear more frequently while still preserving the procedural nature of the output.

Breadth-First Search

Grid-based BFS pathfinding that guarantees the shortest path in an unweighted space. It uses a FIFO queue to explore tiles level-by-level, ensuring all possible routes are checked in increasing distance from the start. To ensure the absolute shortest path possible, I added a sort for each set of neighbours by their squared magnitude distance to the target before queuing them, so the closest direction always gets explored first.

Depth-First Search

Depth-first search (DFS) pathfinding with randomization, using a LIFO stack where each node pushes its neighbours so the most recently discovered direction is always explored next. Each node pushes its neighbours onto the stack, meaning the most recently discovered direction is always explored next, resulting in long, continuous paths before the algorithm branches elsewhere. To avoid rigid or repetitive routes, I randomized the neighbour order before pushing them onto the stack. This ensures the search direction varies each run, producing more natural, organic-looking pathing instead of predictable "snaking" patterns caused by fixed directional bias.

// deep_dive - Adding pathfinding with multiple path support
problem ◎
Path lists don't cleanly connect

The end of one path and the start of the next were often more than 1 unit apart. The collapse function expects every neighbour to be exactly 1 unit away, so any larger gap left a tile with no valid solution - resulting in a broken, uncollapsable tile at the junction.

approach ◈
Adding an additional validation layer

The solution was adding an additional constraint; instead of collapsing the lists directly, I first build a dictionary of every gridpoint and its valid connections. 'IsFaceNeighbour' validates that each tile is exactly 1 unit from its neighbours before it gets added. 'AddConnections' then loops through all paths to populate the dictionary, using a hashset per entry to avoid duplicate connections. Only then does the collapse run, using each tile's connection count to pick the right tile type.

result ▸
Validated Junction Resolution

Paths of any count can now be combined and collapsed without broken junctions. Each tile resolves based on its connection count-tiles with one connection become ends, two form straight paths, and three or more create junctions. Overlapping paths can now emerge naturally without any hardcoded rules.

// takeaways[]
▸
Deeper learning through integration
Working with multiple algorithms together, rather than studying each in isolation, forced a deeper understanding of how each works because of the modifications required for them to work in tandem.
▸
The power of constraints
Just like in game design, a small set of constraints can go a long way. Procedural generation takes that idea and runs with it, turning simple rules into a huge range of outcomes.
▸
A worthwhile investment
There's a steep learning curve getting into algorithmic thinking, but the payoff is 100% worth it. This project showed me that algorithms aren't just a single solution - they have the potential to produce a wide variety of results.
